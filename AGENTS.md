# svg2pptx 项目级 Codex 约束

本文件用于约束 Codex 在 `svg2pptx` 仓库内的默认行为。

## 1. Python 环境

- 本项目默认 Python 环境：`conda activate svg2pptx`
- 运行命令、测试、脚本时，默认使用该环境。
- 如果不能直接激活 shell，会等价使用：
  - `conda run -n svg2pptx <command>`

## 2. 依赖管理

- 后续如果需要新增、删除或调整 Python 依赖，统一通过 `requirements.txt` 管理。
- 不要只在本地环境里手动安装而不落盘。
- 如果某次改动引入了新依赖，必须同步更新 `requirements.txt`。

## 3. 修改边界

- 只允许修改当前仓库 `svg2pptx` 内的文件。
- 明确禁止改动任何 `OceanPPT` 项目的代码、配置、测试、脚本和文档。
- 如果需要参考 OceanPPT 的产物或样本，优先使用已经复制到当前仓库内的副本。
- 当前可直接使用的 SVG 样本目录：
  - `.agents/plan/svg2pptx-fork-roadmap/svg_pages/`

## 4. 工作原则

- 先在当前仓库内完成定位、实现和验证，不把改动扩散到外部仓库。
- 涉及环境或依赖变更时，优先保证规则可追溯、可复现。
- 如果某项任务看起来需要修改 OceanPPT 才能完成，先停止，并在当前仓库范围内给出替代方案或阻塞说明。

## 5. 验证要求

- 完成代码改动后，尽量在 `svg2pptx` 环境中运行相关测试或脚本。
- 不能验证时，要明确说明未验证项和原因，不能直接声称“已完成”。

## 6. 代码规则

### 通用原则

- **KISS / YAGNI**：简单直接，不为假设需求过度设计
- **单一职责**：函数做一件事，嵌套控制在 3 层内
- **向后兼容**：未经批准不破坏现有 API/CLI 行为/数据格式
- **复用模式**：按项目既有风格实现，不引入新架构

### ✅ Git 提交策略

- “验证通过”至少包括：
  - 已执行并通过当前任务约定的验收命令
  - 相关测试通过；若项目存在 lint/构建检查，则对应检查通过
  - 本次任务涉及的计划步骤已完成
- 自动提交前必须确认：
  - `git status --short` 中仅包含当前任务相关改动
  - 不存在用户未说明的无关脏文件
  - 本次提交边界清晰，commit 只包含当前任务相关文件
- 以下情况禁止自动提交，直接忽略相关文件即可：
  - 存在无关改动或提交边界不清
  - 测试/lint/构建无法执行或结果不明确
  - 需要 push、amend 或改写历史
- commit message 使用约定式提交

### 语言约定

- 默认使用中文回答，可混用英文技术术语
- 代码标识符（变量、函数、类）使用英文
